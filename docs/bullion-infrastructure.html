Goal & architecture

Run clusters in more than one cloud and have them behave as one active-active platform — not primary + cold backup. A workload deployed to both clouds is one global service; requests load-balance across both at once; if a cloud fails, traffic auto-redistributes to the survivor. All infrastructure-as-code + CI/CD, keyless auth, gated approvals.

CI/CD & security

All deploys run through GitHub Actions with keyless OIDC auth and a gated production environment (required reviewer). No static cloud keys.

Keyless cloud auth

AWS: GitHub OIDC → IAM role fleet-mini-gha. GCP: OIDC → Workload Identity Federation → SA fleet-mini-gha. No long-lived keys.

State & secrets

S3 native state locking (no DynamoDB). Cloud auth is keyless OIDC; only the non-sensitive OIDC identifiers are stored, as GitHub Actions secrets/variables (set via iac/generate-credentials.sh using the gh CLI).

4

Service mesh — final state

The two clusters form one mesh (meshID = mesh-fleet) via Istio multi-primary, multi-network.

How they're joined

  • Shared root CA → workloads in EKS & GKE trust each other's mTLS.
  • A primary istiod per cluster — no control-plane SPOF.
  • East-west gateways (LB) carry cross-cluster mTLS traffic.
  • Remote secrets exchanged → each istiod discovers the other's endpoints.
  • Network labels topology.istio.io/network (network-aws / network-gcp) — the key to cross-network discovery.

Global service & LB

hello.demo.svc.cluster.local runs in both clusters. A DestinationRule sets an explicit 50/50 cross-region split with outlierDetection so a failed cloud is ejected and traffic redistributes (active-active failover).
