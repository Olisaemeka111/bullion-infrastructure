Executive summary

What this is. Bullion Infrastructure is a production-style platform that runs Kubernetes in two public clouds — Amazon EKS and Google GKE — and makes them behave as one active-active cluster. A single application, deployed to both clouds, is served as one global service with traffic load-balanced across both providers simultaneously; if one cloud degrades, traffic automatically shifts to the other.

Why it matters. It removes single-cloud dependency: the platform survives a whole-cloud outage, avoids vendor lock-in, and lets workloads run close to users in multiple regions — without operators managing failover by hand.

The whole platform is reproducible from Git and deployed through gated GitHub Actions pipelines with keyless cloud authentication.

Delivered

  • Two managed Kubernetes clusters (EKS + GKE), provisioned by Terraform.
  • An Istio multi-primary service mesh joining both clusters across clouds (mTLS).
  • Active-active load balancing of a global service across both clouds, with automatic failover (outlier ejection).
  • A cross-cloud private network (AWS↔GCP HA VPN + BGP, 4 tunnels) — live.
  • A central distributed database (CockroachDB multi-region, 4 nodes across both clouds over the VPN), deployed via CI/CD.
  • A real global active-active application (the board game) built from source to GHCR and served from both clouds behind public ingress LoadBalancers.
  • Full CI/CD (GitHub Actions) with keyless OIDC auth and approval gates.
  • A DevSecOps security gate on every push/PR — SAST (CodeQL), secret scanning (Gitleaks), dependency/vuln scanning (Trivy, OWASP Dependency-Check) and IaC scanning (Checkov, tfsec, KICS), plus Ruff linting.
  • Centralised observability (Prometheus/Grafana).

On the roadmap

  • In-cluster control-plane/dashboard (the agentic reconciler running on-cluster).
  • A single global DNS endpoint (one hostname) fronting both ingresses — optional.
  • Production hardening: CockroachDB TLS, least-privilege IAM, private GKE endpoint.
Commercial note. The environment currently runs on a GCP free trial and a pay-as-you-go AWS account for validation. Production hardening (least- privilege IAM, private API endpoints, quota uplifts, the data layer) is scoped in §14/§16.

Business context & objectives

Objectives

  • Resilience: survive a full cloud-provider outage.
  • No lock-in: one platform, portable across clouds.
  • Active-active: use all capacity, not idle standby.
  • Automation: everything reproducible via IaC + CI/CD.
  • Security by default: keyless auth, mTLS, gated changes.

Non-goals (this phase)

  • Production-grade least-privilege IAM (currently broad for validation).
  • Stateful workloads at scale (data layer is on the roadmap).
  • Azure (no subscription available; cleanly disabled, re-enableable).

Design principles

PrincipleHow it's realised
Two control planes, one fabricEach cluster keeps its own K8s + istiod; the mesh + global LB make them one logical platform (you never stretch one control plane across clouds).
Declarative & reproducibleTerraform for infra; manifests for the mesh; GitHub Actions to apply — all in Git.
Keyless & gatedGitHub OIDC to each cloud; production environment requires human approval.
Active-activeGlobal service spans both clusters; DestinationRule splits 50/50; outlier detection fails over.

Solution architecture

East-west (service-to-service)

Istio multi-primary: each cluster's Envoy can reach the other cluster's pods via its east-west gateway over mTLS, using endpoints discovered through exchanged remote secrets.

Delivery

Terraform builds the clusters; GitHub Actions applies infra (deploy.yml) and the platform layer (platform.yml). Every change is planned, reviewed and approved before it reaches production.

CI/CD & automation

PipelineResponsibilityTrigger / gate
ci.ymlRuff lint + 80 unit/integration tests (stdlib) + terraform fmt/validateevery push/PR · no cloud creds
security.ymlDevSecOps scans: CodeQL, Gitleaks, Trivy, OWASP Dependency-Check, Checkov, tfsec, KICSevery push/PR · gate on CRITICAL/HIGH
deploy.ymlTerraform infra (EKS, GKE, VPN): plan → apply → destroyPR=plan · merge=apply (gated) · dispatch=destroy
platform.ymlIstio join + global app + observability across both clustersdispatch / mesh changes (gated)
database.yml · game.ymlCockroachDB (both clusters) · board game (build → GHCR → both clusters)dispatch / path changes

Flow

Open PR → lint + tests + security scans run → automated plan posted as a comment → review → merge → apply pauses for approval in the production environment → resources provision.

Keyless auth

AWS via IAM role + GitHub OIDC; GCP via Workload Identity Federation; tokens are minted per-run and short-lived. No static cloud keys live in GitHub.

Shift-left security gate (security.yml)

Every push/PR runs, with no cloud credentials, gating on CRITICAL/HIGH and any secret: SAST = CodeQL (Python) · Secrets = Gitleaks (full history) · Vulns/deps = Trivy + OWASP Dependency-Check (CVSS ≥ 7) · IaC = Checkov + tfsec + KICS over iac/terraform. Findings upload to the Security tab as SARIF. CodeQL and Gitleaks pass clean; the IaC and dependency scanners surface real Terraform hardening items for triage.

Security & compliance

Identity & access

  • GitHub OIDC → AWS IAM role / GCP WIF (no static keys).
  • EKS access entries map identities to cluster RBAC.
  • GKE Workload Identity for pod-level GCP access.
  • Production deploys require human approval (GitHub environment).

In-transit & supply chain

  • Mesh-wide mTLS (shared CA) for service-to-service traffic.
  • Reference admission model: signed images from trusted registries, PSS.
  • Remote state encrypted in S3 with locking.
Known hardening items (validation environment). The bootstrap AWS key was shared during setup and must be rotated; the operator IAM user currently holds AdministratorAccess (scope down); the GKE control-plane endpoint is publicly reachable (restrict via authorized networks); and CockroachDB currently runs in insecure mode (enable TLS for production). Deploy identifiers are held as GitHub Actions secrets; for runtime app secrets a cloud-native secret manager + External Secrets is recommended.

Data layer — CockroachDB (deployed)

Active-active traffic needs active-active data. A distributed multi-master database lets every cloud read and write and survive a region loss.
AspectImplementation
EngineCockroachDB v24.3.5 as a DaemonSet (one node per K8s node) joined into one logical DB
NetworkinghostNetwork — each node advertises its routable node IP (pod IPs aren't routable cross-cloud)
Connectivitynode-to-node (:26257) over the AWS↔GCP HA VPN (cross_cloud module); --join built from live node IPs
DeploymentCI/CD via database.yml (discovers node IPs → applies to both clusters → init once)
StatusLIVE — 4 nodes across both clouds (2 EKS + 2 GKE)
Hardening. The cross-cloud VPN is up and the cluster is live in insecure mode for this validation; enabling TLS (cert-manager / cockroach cert) is the production-hardening step.

Reference control plane (offline model)

The repository also contains an agent-driven control-plane reference implementation (Python) that models the full lifecycle — reconciliation loop, provider abstraction, node agent, security gate, networking fabric, observability — and runs entirely offline with 80 passing tests. It proves the design (including the active-active traffic model in networking/global_lb.py + sim/active_active.py) deterministically.

On the live managed clusters, its responsibilities (provision / scale / self-heal) are met by Terraform + managed node groups; the Provider interface is the documented seam for driving real clouds directly in a future phase. An in-cluster deployment of this control plane + dashboard is on the roadmap.

Access

gcloud container clusters get-credentials bullion-gke --zone us-central1-a --project project-f38bfcd6-70ff-4a8d-9cd

Deploy / change

Open a PR (auto plan) → review → merge → approve the gated apply. Platform layer: Actions → platform → Run workflow → approve.

Verify active-active

kubectl --context eks -n demo run curl --image=curlimages/curl --restart=Never --command -- sleep 3600
kubectl --context eks -n demo exec curl -c curl -- sh -c \
  "for i in 1 2 3 4 5 6 7 8; do curl -s hello.demo.svc.cluster.local; echo; done"
# expect a mix of "hello from aws" and "hello from gcp"

Failure handling

A cloud/endpoint failure is ejected by outlierDetection and traffic shifts to the survivor automatically. No manual failover is required for the traffic layer.

Teardown (stop costs)

Actions → deploy → Run workflow → action=destroy → approve
